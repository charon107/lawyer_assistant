# AI Agent 在金融法律领域的应用研究计划（小型律所·阿里云起步·最低预算）

## 目标
针对**小型律所（5-10人）**，聚焦**资本市场、信托、基金**等金融法律领域，以**最小预算**在阿里云 ECS 起步（后续可迁移至本地），产出详尽的 AI Agent 技术分析与落地方案。

## 用户约束
- **规模**：小型律所，5-10 名律师
- **领域**：金融法律 — 资本市场（IPO/再融资/并购重组）、信托、基金（公募/私募合规）
- **预算**：尽可能低，阿里云 ECS 起步 → 后续本地化
- **产出**：纯技术分析，无需部署脚本

## 一、金融法律领域的 AI Agent 核心场景

### 1.1 场景优先级矩阵（按价值×可行性排序）

| 优先级 | 场景 | 痛点 | AI Agent 角色 | 技术可行性 |
|--------|------|------|---------------|-----------|
| P0 | **合同/协议审查** | 基金合同、信托协议、LP协议条款繁杂，人工比对耗时 | 自动提取关键条款、风险标记、版本对比 | ⭐⭐⭐⭐⭐ |
| P0 | **法规合规检索** | 证监会/基协/银保监会规定频繁更新，人工跟踪困难 | 法规变更监控、合规差距分析 | ⭐⭐⭐⭐⭐ |
| P1 | **尽职调查辅助** | IPO/并购中大量尽调材料审查 | 自动抽取关键信息、异常标记 | ⭐⭐⭐⭐ |
| P1 | **法律意见书起草** | 标准化法律意见书模板填充与校验 | 模板管理、条款自动生成、一致性检查 | ⭐⭐⭐⭐ |
| P2 | **判例/处罚案例检索** | 证监会处罚、法院判例检索与分析 | 语义检索 + 摘要生成 | ⭐⭐⭐⭐ |
| P2 | **监管问询回复辅助** | 交易所问询函回复起草 | 历史回复检索、要点提取 | ⭐⭐⭐ |

### 1.2 金融法律文档的特殊性
- **文档长度**：基金合同/招股说明书通常 100-500 页 → 需要长上下文模型（128K+ tokens）
- **表格密集**：财务数据、费率表、分配瀑布 → 需要表格解析 + 结构化提取
- **术语专业**：LPA、PPM、side letter、waterfall、carried interest → 需要金融法律专用知识库
- **交叉引用**：协议内大量「参见第X条」「根据XX法规」→ 需要引用追溯

### 2.3 RAG（检索增强生成）架构

```
┌────────────────────────────────────────────┐
│              文档处理管道                    │
│  PDF/Word → OCR → 段落分块 → Embedding     │
└───────────────┬────────────────────────────┘
                │
┌───────────────▼────────────────────────────┐
│            向量数据库（轻量级）               │
│  ChromaDB / LanceDB（单机零配置）            │
│  Qdrant（生产推荐，Docker 一键部署）         │
└───────────────┬────────────────────────────┘
                │
┌───────────────▼────────────────────────────┐
│           检索 + 重排序                      │
│  BGE-M3 Embedding + BGE-Reranker-v2        │
└───────────────┬────────────────────────────┘
                │
┌───────────────▼────────────────────────────┐
│           LLM 推理（生成回答）               │
│  DeepSeek API / 本地 Qwen2.5 / Ollama      │
└────────────────────────────────────────────┘
```

**分块策略（金融法律文档专用）**：
- 按条款/章节语义分块（非固定长度）
- 每个条款保留上下文标题路径（如：「第3章→第3.2条→(a)款」）
- 表格单独提取为结构化 JSON
- 分块大小：512-1024 tokens（保证检索精度）

**推荐向量模型**：
- BGE-M3（多语言，567M 参数，CPU 可运行）
- BGE-large-zh-v1.5（纯中文，326M 参数，速度快）

### 2.4 Agent 框架选择

| 框架 | 适合场景 | 学习曲线 | 部署难度 |

## 四、金融法律 RAG 知识库构建

### 4.1 数据源清单

| 类别 | 数据源 | 获取方式 | 预估量 |
|------|--------|---------|--------|

| **招股书** | 港交所/上交所/深交所披露 | 公开下载 | 按需获取 |
| **行业研究** | 基金业协会年报、行业分析 | 公开获取 | 100+ 篇 |

### 4.2 文档处理管道

```python
# 处理流程伪代码
PDF/Word 输入
  → MinerU 解析（PDF→Markdown，保留表格）
  → 页面级分块（最大 1024 tokens）
  → 提取元数据（文档类型、发布日期、文号、来源）
  → BGE-M3 Embedding
  → 存入 ChromaDB/Qdrant（带元数据过滤）
```

### 4.3 检索策略

```
用户提问 → 关键词提取 → 混合检索
                        ├── 语义检索（Embedding 相似度）
                        └── 关键词检索（BM25）
                              ↓
                        ReRanker 重排序
                              ↓
                        Top-K 上下文注入 LLM
                              ↓
                        生成回答 + 引用溯源
```

## 五、金融法律 Agent 核心工作流设计

### 5.1 合同审查 Agent（P0）

```
输入：基金合同 / 信托协议 / LPA
  ↓
[解析] PDF→结构化文本，识别条款
  ↓
[提取] 关键条款：管理费、carry、分配瀑布、LP权利、
        GP义务、赎回条款、清算优先级
  ↓
[对比] 与标准模板 / 历史版本对比
  ↓
[风险标记] 异常条款、缺失条款、模糊用语
  ↓
[输出] 审查报告 Markdown + 风险评分
```

### 5.2 法规合规 Agent（P0）

```
触发：定时任务 / 手动查询
  ↓
[监控] RSS/API 获取最新法规 → 变更检测
  ↓
[分析] 新旧法规对比 → 差异提取
  ↓
[匹配] 与所内业务/客户基金产品关联
  ↓
[建议] 合规差距分析 → 整改建议
  ↓
[输出] 合规简报（每日/每周）
```

### 5.3 尽调辅助 Agent（P1）

```
输入：尽调材料包（PDF/Excel/扫描件）
  ↓
[解析] OCR + 表格提取
  ↓
[分类] 工商信息、股权结构、财务数据、关联交易、
       诉讼仲裁、知识产权
  ↓
[交叉验证] 多源数据一致性检查
  ↓
[异常标记] 数据矛盾、重大变化、缺失信息
  ↓
[输出] 尽调摘要 + 异常清单
```

## 七、关键风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| **数据安全** | 客户合同/尽调材料上云有合规风险 | 本地化部署，云端仅用 API（不传文档） |
| **模型幻觉** | 生成虚构法条/判例 | RAG 强制引用原文，输出校验 Agent |
| **金融术语歧义** | 模型对 carry/waterfall 理解错误 | 术语表 + Few-shot 示例 |
| **长文档截断** | 500 页招股书超出上下文 | 分章节处理 + Map-Reduce 汇总 |
| **监管不确定性** | 律师使用 AI 的执业规范未明确 | 保留人工审核环节，AI 仅做辅助 |
